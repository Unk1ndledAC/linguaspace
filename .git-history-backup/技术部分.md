# 语界 LinguaSpace 技术架构与开发方案

## 1. 系统建设目标

语界系统面向南亚、东南亚游客，建设一个支持**多语交互、智能讲解、拍照识别、真人导游协同和AI导游实训**的文旅智能服务平台。

系统核心不是单纯接入大模型，而是将开源预训练模型与云南文旅知识库结合，形成一个可检索、可审核、可持续扩展的文旅AI系统。

整体技术路线为：

```text
多模态输入
语音 / 文字 / 图片 / 位置
        ↓
AI能力处理
语音识别 / 翻译 / 图像理解 / 大模型生成
        ↓
知识增强
RAG文旅知识库 / 文化知识图谱
        ↓
业务输出
多语导览 / 拍照问答 / 路线推荐 / 实训评分 / 导游协同
```

---

# 2. 总体技术架构

系统采用分层架构，主要分为五层：

```text
客户端层
游客端 / 学生端 / 导游端 / 管理端

业务服务层
智能导览服务 / 实训服务 / 导游协同服务 / 知识库管理服务

AI能力层
语音识别 / 机器翻译 / 图像理解 / 大语言模型 / 语音合成

知识增强层
文旅知识库 / 向量检索 / 文化知识图谱 / RAG生成

数据与运维层
业务数据库 / 向量数据库 / 图数据库 / 文件存储 / 日志监控
```

## 2.1 客户端层

客户端包括四类入口：

| 端口  | 主要功能                    |
| --- | ----------------------- |
| 游客端 | 语音提问、文字提问、拍照识别、多语导览     |
| 学生端 | 虚拟带团训练、语音讲解提交、训练报告查看    |
| 导游端 | 查看游客问题、接管AI对话、修正AI讲解内容  |
| 管理端 | 知识库管理、术语表维护、内容审核、模型效果查看 |

游客端可优先采用H5或小程序实现，管理端和学生端建议采用Web端实现。

# 3. 后端服务架构

后端建议采用模块化服务设计，前期可使用单体模块化开发，后期根据并发量拆分为微服务。

| 日志服务   | 记录模型调用、检索结果、错误与反馈       |

# 4. AI能力层设计

系统AI能力由多个开源模型组成，不建议使用单一大模型完成所有任务。

## 4.1 模型选型

| 能力    | 推荐模型                     | 作用             |
| ----- | ------------------------ | -------------- |
| 大语言模型 | Qwen3-14B-Instruct       | 问答理解、导览生成、实训评分 |
| 语音识别  | Whisper / faster-whisper | 游客语音转文字        |
| 翻译模型  | NLLB-200                 | 中文与东南亚、南亚语种互译  |
| 图像理解  | Qwen2.5-VL-7B-Instruct   | 景点、文物、菜单、标识牌识别 |
| 向量模型  | BGE-M3                   | 多语知识检索         |
| 语音合成  | Piper / MeloTTS          | 文本回答转语音播报      |

## 4.2 AI调用链

智能导览的典型调用链如下：

```text
游客输入
  ↓
如果是语音：Whisper 转写
如果是图片：Qwen2.5-VL 识别
如果是文本：直接处理
  ↓
NLLB-200 翻译为系统统一语言
  ↓
BGE-M3 生成向量
  ↓
Milvus 检索文旅知识库
  ↓
Neo4j 补充文化关系
  ↓
Qwen3 基于检索结果生成回答
  ↓
NLLB-200 翻译为游客目标语言
  ↓
TTS生成语音，可选
  ↓
返回游客端
```

这里需要注意：**大模型不直接回答文化类问题，必须先经过知识库检索。**

# 5. RAG文旅知识库设计

RAG是系统核心模块，用于解决大模型回答不稳定、容易编造的问题。

## 5.1 知识库内容

知识库应包含：

| 类型   | 内容示例              |
| ---- | ----------------- |
| 景点介绍 | 大理古城、丽江古城、石林、西双版纳 |
| 民族文化 | 白族、傣族、纳西族、彝族文化    |
| 非遗项目 | 东巴文字、白族扎染、纳西古乐    |
| 节庆习俗 | 泼水节、火把节、三月街       |
| 饮食文化 | 过桥米线、傣味、鲜花饼       |
| 建筑文化 | 白族民居、傣族干栏式建筑      |
| 路线推荐 | 亲子路线、研学路线、摄影路线    |
| 禁忌礼仪 | 宗教禁忌、拍照禁忌、民族习俗提醒  |
| 实训案例 | 导游讲解案例、游客常见问题     |

## 5.2 入库流程

```text
原始资料收集
文旅局资料 / 景区资料 / 导游词 / 高校课程资料 / 非遗资料
        ↓
文本清洗
去重、去无关内容、统一格式
        ↓
知识切分
按景点、主题、段落、问答对切分
        ↓
人工审核
确认内容准确、可公开使用
        ↓
向量化
使用 BGE-M3 生成向量
        ↓
写入数据库
PostgreSQL 保存文本，Milvus 保存向量
        ↓
图谱抽取
抽取景点、民族、节庆、非遗等关系
        ↓
写入 Neo4j
```

## 5.3 检索规则

检索时建议采用：

```text
问题向量化
  ↓
Milvus召回相关知识片段
  ↓
按地点、景点、主题、审核状态过滤
  ↓
选择最相关的若干条知识
  ↓
与图谱关系一起送入大模型
```

需要强制过滤：

1. 未审核内容不进入正式回答；
2. 敏感内容必须经过人工确认；
3. 景区动态信息需要标注更新时间；
4. 找不到可信资料时，系统应提示“暂无可靠资料”，而不是让模型自由发挥。

# 6. 文化知识图谱设计

知识图谱用于表达文旅内容之间的关系，弥补普通文本检索只找“相似段落”、但不理解文化关系的问题。

## 6.1 图谱实体

| 实体类型 | 示例             |
| ---- | -------------- |
| 景点   | 大理古城、丽江古城、石林   |
| 城市   | 昆明、大理、丽江、西双版纳  |
| 民族   | 白族、傣族、纳西族、彝族   |
| 节庆   | 泼水节、火把节、三月街    |
| 非遗   | 白族扎染、东巴文字、纳西古乐 |
| 建筑   | 白族民居、傣族竹楼      |
| 饮食   | 过桥米线、傣味、鲜花饼    |
| 禁忌   | 寺庙礼仪、拍照禁忌、饮食禁忌 |

## 6.2 图谱关系

```text
景点 → 位于 → 城市
景点 → 关联 → 民族
民族 → 拥有 → 节庆
民族 → 拥有 → 非遗
民族 → 拥有 → 建筑形式
景点 → 推荐 → 游览路线
场所 → 存在 → 礼仪禁忌
```

示例：

```text
大理古城 → 关联 → 白族
白族 → 拥有 → 三道茶
白族 → 拥有 → 白族扎染
白族 → 拥有 → 白族民居
白族民居 → 常见元素 → 照壁
```

## 6.3 使用方式

当游客问：

```text
大理古城和白族文化有什么关系？
```

系统先从知识库检索“大理古城”“白族文化”等文本内容，再从图谱补充：

```text
大理古城
  ├── 白族
  ├── 三道茶
  ├── 白族扎染
  ├── 白族民居
  └── 三月街
```

然后由大模型组织成自然讲解。

# 7. 核心业务数据流

## 7.1 文本问答流程

```text
游客输入问题
  ↓
智能导览服务接收
  ↓
翻译为中文或系统统一语言
  ↓
RAG检索知识库
  ↓
图谱补充关系
  ↓
大模型生成回答
  ↓
翻译为游客语言
  ↓
返回游客端
```

## 7.2 语音问答流程

```text
游客录音
  ↓
上传音频
  ↓
Whisper语音识别
  ↓
得到文本问题
  ↓
进入文本问答流程
  ↓
生成回答
  ↓
TTS合成语音
  ↓
游客端播放
```

## 7.3 图片问答流程

```text
游客上传图片
  ↓
视觉模型识别图片内容
  ↓
提取景点、物体、文字、菜单、标识牌等信息
  ↓
生成检索关键词
  ↓
进入RAG检索流程
  ↓
生成讲解或翻译结果
  ↓
返回游客端
```

## 7.4 AI导游实训流程

```text
学生选择训练场景
  ↓
系统生成虚拟游客角色
  ↓
虚拟游客提出问题
  ↓
学生语音回答
  ↓
ASR转写
  ↓
知识库比对
  ↓
大模型按维度评分
  ↓
生成训练报告
```

# 8. AI导游实训模块设计

实训模块面向高校导游人才培养，用于模拟真实带团过程。

## 8.1 训练场景

可设计以下类型：

| 场景     | 说明                |
| ------ | ----------------- |
| 景点讲解   | 学生介绍某一景点          |
| 民族文化讲解 | 学生解释民族习俗、节庆、饮食    |
| 接团服务   | 模拟游客到达、集合、行程说明    |
| 投诉处理   | 模拟游客不满、路线变更、服务争议  |
| 跨文化交流  | 模拟不同国家游客的文化问题     |
| 应急处理   | 模拟迷路、天气变化、身体不适等情况 |

## 8.2 评分维度

评分不只给总分，应拆成多个维度：

| 维度     | 说明             |
| ------ | -------------- |
| 内容准确性  | 是否符合知识库标准内容    |
| 讲解完整度  | 是否覆盖关键知识点      |
| 表达清晰度  | 语言是否流畅、有逻辑     |
| 服务礼仪   | 是否符合导游服务规范     |
| 应变能力   | 是否能处理追问和突发问题   |
| 跨文化敏感性 | 是否避免误解、冒犯或禁忌表达 |
| 多语能力   | 外语表达是否准确、自然    |

## 8.3 数据沉淀

实训数据可沉淀为三类资源：

1. 学生训练记录；
2. 优秀讲解案例；
3. 常见错误案例。

优秀案例经过人工审核后，可进入案例库，用于后续教学和AI生成参考。

# 9. 真人导游协同模块设计

该模块用于连接AI导览和真人导游。

## 9.1 AI负责

1. 高频问题回答；
2. 多语基础讲解；
3. 实时翻译；
4. 拍照识别；
5. 路线推荐；
6. 游客兴趣摘要。

## 9.2 真人导游负责

1. 深度文化解释；
2. 复杂问题回答；
3. 民族、宗教、历史内容把关；
4. 个性化服务；
5. 突发情况处理；
6. 修正AI回答。

## 9.3 协同流程

```text
游客与AI交互
  ↓
系统记录问题、兴趣、语言、地点
  ↓
生成游客会话摘要
  ↓
真人导游查看摘要
  ↓
必要时接管对话
  ↓
导游修正AI回答
  ↓
修正内容进入审核池
  ↓
审核通过后进入知识库或案例库
```

# 11. 开发阶段安排

## 第一阶段：文本版RAG导览

目标：先跑通知识库问答。

开发内容：

1. 管理端上传文旅资料；
2. 文档切分与审核；
3. 向量化入库；
4. 文本问答；
5. RAG检索；
6. 大模型生成回答；
7. 返回知识来源。

完成后系统应能回答：

```text
大理古城有什么历史？
白族三道茶是什么意思？
西双版纳和傣族文化有什么关系？
```

---

## 第二阶段：语音与翻译

目标：实现多语语音问答。

开发内容：

1. 音频上传；
2. Whisper语音识别；
3. NLLB-200翻译；
4. 多语回答；
5. TTS语音播报；
6. 文旅术语表。

完成后游客可以用语音提问，系统用对应语言回答。

---

## 第三阶段：图片识别

目标：实现“拍照问问题”。

开发内容：

1. 图片上传；
2. 图像理解；
3. OCR识别；
4. 图片内容生成检索关键词；
5. 图片问答接入RAG。

完成后游客可以拍摄建筑、菜单、指示牌、民族服饰等内容并提问。

---

## 第四阶段：文化知识图谱

目标：增强文化关联解释能力。

开发内容：

完成后系统可以回答更复杂的文化关联问题。

---

## 第五阶段：AI导游实训

目标：实现学生虚拟带团训练。

开发内容：

1. 实训场景管理；
2. 虚拟游客生成；
3. 学生语音回答；
4. 讲解内容比对；
5. 多维度评分；
6. 训练报告生成。

---

## 第六阶段：真人导游协同

目标：实现AI与真人导游联动。

开发内容：

1. 导游端；
2. 游客会话摘要；

# 12. 开发重点与约束

## 12.1 大模型必须走知识库

涉及以下问题时，不允许大模型直接回答：

```text
景点历史
民族文化
宗教习俗
非遗项目
禁忌礼仪
政策信息
开放时间
票价信息
```

必须先检索知识库。如果知识库没有内容，应返回“暂无可靠资料”，而不是让模型编造。

---

## 12.2 回答必须可追踪

每次AI回答至少保存：

```text
用户问题
标准化问题
检索到的知识片段
使用的大模型版本
回答内容
是否触发人工审核
游客反馈
```

这样后期才能排查错误回答、优化知识库和模型提示词。

---

## 12.3 术语必须统一

以下内容需要进入术语表：

```text
景点名称
民族名称
节庆名称
非遗项目
饮食名称
宗教礼仪
旅游服务词汇
```

避免同一个词在不同语言中出现多种翻译。

---

## 12.4 图片识别不能直接替代知识库

视觉模型只负责识别图片内容，不能直接生成文化解释。

正确流程是：

```text
图片识别
  ↓
生成关键词
  ↓
检索知识库
  ↓
大模型基于知识库回答
```

---

# 13. 最小可用版本 MVP

MVP建议优先完成以下功能：

1. 游客文本问答；
2. 游客语音问答；
3. 中文、英文、越南语、泰语、印尼语基础支持；
4. 3个示范景区知识库；
5. RAG检索；
6. AI回答来源记录；
7. 管理端知识上传与审核；
8. 简单AI导游实训；
9. 学生语音转写和评分；
10. 导游修正内容入库。

MVP阶段可以暂缓：

1. 完整知识图谱；
2. 所有小语种语音合成；
3. 高并发部署；
4. 复杂路线规划；
5. 完整真人导游接管流程。

---

# 14. 总结

语界系统的工程实现可以概括为：

```text
前端采集游客输入
        ↓
后端统一编排AI能力
        ↓
语音、翻译、视觉模型完成输入理解
        ↓
RAG知识库和文化知识图谱提供可靠依据
        ↓
大语言模型生成讲解
        ↓
结果返回游客端、实训端或导游端
        ↓
人工审核和用户反馈反向更新知识库
```

开发时应优先保证三件事：

1. **知识库准确**：文旅内容先审核再使用；
2. **调用链清晰**：语音、翻译、检索、大模型各模块职责明确；
3. **结果可追踪**：每次回答都能查到来源、模型版本和处理过程。
