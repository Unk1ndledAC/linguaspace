# 语界 LinguaSpace 文档与项目一致性检验报告

> 检验日期：2026-06-02
> 检验范围：`docs/文档/` 中的开发文档（需求分析、系统设计、测试文档、用户手册）vs `frontend/` 和 `backend/` 实际实现

---

## 一、总体结论

文档与实际项目之间存在**多处显著不一致**，涉及前端技术栈、后端数据库选型、向量数据库、模型服务、API 路由设计、页面结构和子系统职责划分。文档反映的是"规划中的设计"，而实际项目是"已落地的实现"，两者在多个关键维度上产生了偏差。建议按本报告逐项修正文档，以确保文档作为项目交付物的可追溯性和可参考性。

## 三、前端页面结构不一致

### 3.1 文档描述 vs 实际 `navigation.ts`

文档（需求分析第七节"每种用户的用例"）定义了四类用户的用例：
- **游客用例**：文本提问、语音提问、图片提问、查看多语讲解、获取路线推荐、提交反馈
- **学生用例**：选择实训场景、与虚拟游客对话、提交语音讲解、查看训练报告、查看训练历史
- **导游用例**：查看游客会话摘要、接管游客对话、修正 AI 回答、提交优质讲解案例、查看接管记录
- **管理员用例**：管理用户与权限、上传知识文档、审核知识内容、维护术语表、维护知识图谱、查看模型调用日志、管理用户反馈

实际 `navigation.ts` 中的模块和页面划分：

| 文档角色 | 文档页面 | 实际模块 | 实际页面 |
|---|---|---|---|
| 游客 | 文本/语音/图片/多语讲解/路线/反馈 | tourist | home / chat / voice / image / routes / culture-tips / history |
| **学生** | 实训场景/虚拟游客对话/语音讲解/报告/历史 | **无独立模块** | **缺失** |
| 导游 | 会话摘要/接管/修正/案例/接管记录 | guide | dashboard / sessions / takeover / corrections / cases / profile |
| 管理员 | 用户/知识/审核/术语/图谱/日志/反馈 | knowledge | documents / chunks / graph / review / terms / rag-test / statistics |
| 管理员 | 系统状态 | system | dashboard / users / roles / permissions / health / logs / metrics / settings |

### 3.2 具体不一致项

**问题 1：学生端完全缺失**
文档定义了完整的学生用例（选择实训场景、与虚拟游客对话、提交语音讲解、查看训练报告、查看训练历史），但实际 `navigation.ts` 中既没有 `student` 模块，也没有对应的实训相关页面（训练场景页、讲解提交页、评分报告页、训练历史页全部缺失）。当前前端中导游模块部分承担了实训功能，但文档中的学生用例未被实现。

**修正建议**：在需求分析文档中将 FR-10（训练场景管理）和 FR-11（训练评分报告）的执行者从"学生"修正为"学生/导游"，并在用户手册中明确说明当前版本中学生实训功能通过导游端访问。

**问题 2：文化提醒页面**
文档游客端用例中未提及"文化提醒"功能，但 `navigation.ts` 中存在 `/tourist/culture-tips` 页面，对应接口 `GET /api/tourist/culture-tips`。该功能在文档中无对应用例和需求编号。

**修正建议**：在需求分析文档的游客用例中补充"查看文化提醒"用例，在功能需求表中补充对应需求编号（如 FR-16）。

**问题 3：游客首页拆分**
文档中游客端首页的功能被拆分为 `tourist/home`、`tourist/history` 等独立页面，而文档中的用例未描述"游览历史"功能。

**修正建议**：在需求分析文档游客用例中补充"查看游览历史"用例。

**问题 4：intro 模块**
文档的系统用例中未描述 intro（项目介绍）模块，但实际前端存在完整的 intro 模块（overview / architecture / features / scenarios / roadmap）。该模块承担对外展示和项目介绍功能。

**修正建议**：在需求分析文档的"软件总体流程"或新增"系统展示端"章节中补充对 intro 模块的说明，或在项目定位中明确系统包含对外展示门户。

## 四、技术栈不一致

### 4.1 前端框架

| 文档描述 | 实际实现 |
|---|---|
| Vue 3、H5、小程序 | **React 18 + TypeScript**（`frontend/src/App.tsx` 使用 React） |

**修正建议**：系统设计文档第 97 行"前端框架"一栏，将"Vue 3、H5、小程序"修正为"React 18 + TypeScript + Vite"，并说明管理端和学生端采用 React PC 端方案。

### 4.2 数据库

| 文档描述 | 实际后端 |
|---|---|
| PostgreSQL + pgvector | **MySQL**（`settings.mysql_url`，`health()` 中 `mysql_host_port()`） |

**修正建议**：系统设计文档第 98 行、数据库表设计章节（从第 539 行开始）和数据存储详细设计章节中，将 PostgreSQL 全部替换为 MySQL，并更新表结构设计中的 SQL 方言（UUID 主键策略、TEXT 类型映射等）。

### 4.3 向量数据库

| 文档描述 | 实际实现 |
|---|---|
| Milvus | **无独立 Milvus**，通过 `embeddings.py` 中的 `EmbeddingAdapter` 配合本地向量计算实现 |

**修正建议**：系统设计文档第 99 行、总体技术选型表、数据存储详细设计章节中，将"Milvus"替换为"本地 Embedding 适配器（EmbeddingAdapter）"，说明向量检索通过 BGE-M3 模型在应用层实现，不依赖独立向量数据库服务。

### 4.4 图数据库

| 文档描述 | 实际实现 |
|---|---|
| Neo4j | **graph_mirror（内存图谱）**（`infrastructure.py` 中 `graph_mirror`，`health()` 返回 `"graph_adapter": {"ok": True, "backend": "in-memory"}`） |

**修正建议**：系统设计文档第 100 行和图数据库相关章节中，将 Neo4j 替换为"内存图谱（graph_mirror）"，并说明文化图谱数据以字典结构保存在内存中，支持基本的增删改查操作。

### 4.5 模型服务

| 文档描述 | 实际实现 |
|---|---|
| vLLM / Transformers，Qwen3-14B-Instruct，Qwen2.5-VL | **Ollama**（`health()` 返回 `"llm": {"provider": "ollama"}`，`vision_provider` 使用 Ollama Vision） |

**修正建议**：系统设计文档第 103 行、模型适配接口表（从第 514 行开始）中，将模型推理框架从"vLLM、Transformers"修正为"Ollama 本地模型服务"，Qwen3-14B-Instruct 修正为实际运行的 Ollama 模型名称。

## 五、子系统与模块职责不一致

### 5.1 学生实训子系统

文档（系统设计第 303-357 行）定义了完整的 AI 导游实训子系统，包含训练场景设计、评分维度、Topo-SCTB-Net 增强方案。但实际代码中：
- 没有独立的训练场景管理界面（无 `training/scenarios` 的管理页面）
- 实训评分直接通过 `POST /api/training/score` 提交，无专门的报告生成接口
- Topo-SCTB-Net 在代码中无实现

**修正建议**：在系统设计文档中标注 Topo-SCTB-Net 为"设计提案"而非已实现功能，或在后续迭代中补充实现。

### 5.2 知识入库状态机

文档（系统设计第 267-282 行）定义了完整的状态机：UPLOADED → PARSED → CHUNKED → PENDING_REVIEW → APPROVED → REJECTED → PUBLISHED → OFFLINE。

实际 `runtime.py` 和 `store.py` 中的文档上传和切片流程相对简化，审核流程中不存在文档级别的 `PARSED`、`CHUNKED` 状态（只有 `pending`、`approved`、`rejected`）。

**修正建议**：在系统设计文档知识入库状态机表格中，将状态简化为 `draft`（或 `pending`）→ `approved` → `rejected` → `offline`，或补充文档说明当前 MVP 阶段使用简化状态流转。

### 5.3 用户角色定义

文档定义了四种角色：游客（tourist）、学生（student）、导游（guide）、管理员（admin）。

实际 `runtime.py` 中的角色定义为：`tourist`、`guide`、`admin`（student 角色不存在）。

**修正建议**：在需求分析文档第 34-36 节和系统设计文档第 215-229 行权限矩阵中，将"学生"角色合并到现有角色体系中，或在实现层面补充 student 角色的支持。

## 七、文档内部一致性问题

### 7.1 需求追踪矩阵（requirements.tex 第 823-847 行）

需求追踪矩阵中的接口名称与系统设计文档中的接口表不完全一致。例如需求文档中引用 `POST /api/training/submissions`，但系统设计文档中该接口描述为另一形式，实际后端又不同。应统一为后端实际路由。

### 7.2 系统设计文档接口表（system_design.tex 第 374-398 行）

接口表中的请求参数与 `main.py` 中实际定义的 Pydantic Model 不完全匹配。例如 `ChatRequest` 中实际字段为 `question, language, location, session_id`，而文档接口表中 `POST /api/guide/text` 的参数描述为 `question, language, spotId, session_id`（spotId 在实际代码中不存在）。

### 7.3 数据库表结构（system_design.tex 第 539-750 行）

文档定义了详细的数据库表结构（users、roles、permissions、guide_sessions、messages 等），但实际后端使用 MySQL 而非 PostgreSQL，且表结构通过 `runtime.py` 中的字典操作而非 SQLAlchemy ORM 实现。文档中的 SQL 类型定义（如 `varchar`、`timestamptz`）需要根据 MySQL 方言重新表述。
